# 01. コンテキストウィンドウとセッション運用

> agent-coach 観点 **1（トークン消費 hot spot）** と **4（コンテキストロット）** に対応する基礎章。

## 1.1 大原則: コンテキストはハードな上限

Claude Code のベストプラクティスはほぼすべて「コンテキストウィンドウは有限で、満ちると性能が落ちる」という制約から導かれる（公式 [Best practices](https://code.claude.com/docs/en/best-practices)）。

コンテキストに載るもの:

- 会話履歴（user / assistant / tool_use / tool_result すべて）
- 読み込んだファイル（`@` 参照、Read 結果、貼り付け）
- コマンド出力
- `CLAUDE.md`（プロジェクト + 親 + ホーム + import）
- 自動メモリ（`MEMORY.md` の先頭 200 行 / 25KB）
- ロード済みスキル（description は常駐、本体は呼出時に挿入）
- システムプロンプト
- MCP ツール定義（一部 deferred）

### コンテキストロット（context rot）

LLM 性能はコンテキストが満ちるほど劣化する。アテンションが分散し、古い・無関係なコンテンツが現在のタスクを妨げる。

- 1M トークンモデルでも **30〜40 万トークン付近**から劣化が観測される（[Thariq](https://x.com/trq212/status/2044548257058328723)）
- 200K モデル（Opus 等）では実効上限が **70K 程度**に縮むケースもある（[fabymetal](https://note.com/fabymetal/n/n3f0f2873b56c)）
- 上限到達時に **auto-compaction** が走るが、要約は損失を伴う。**早期にコンパクションさせない設計**が望ましい

### コンテキスト消費の常時計測

公式が推奨する習慣:

- `/context` でセクション別（システム / ファイル / 会話 / MCP / スキル / メモリ）に使用量を可視化
- カスタム status line（`/statusline`）でターン毎にトークン消費を表示
- `/mcp` で MCP サーバ毎のコストを確認
- **MCP ツール定義は deferred 化**されており、ツール検索で必要時のみロードされる（公式仕様）

ガイドライン目安（[fabymetal](https://note.com/fabymetal/n/n3f0f2873b56c)）:

- 設定済み MCP: 20〜30
- 有効化 MCP: 10 以下
- アクティブツール: 80 以下
- 60% 超 → 次セッションを検討

## 1.2 ターン終了時の 5 つの分岐

ターンが終わった瞬間、ユーザの選択肢は 5 通り（[Thariq](https://x.com/trq212/status/2044548257058328723)）:

| 選択 | 操作 | 用途 |
| --- | --- | --- |
| **Continue** | そのまま続ける | 自然だが、コンテキスト管理しないと汚染が進む |
| **Rewind** | `Esc Esc` または `/rewind` | 直近 N ターンを破棄して再プロンプト |
| **Clear** | `/clear` | 学んだことをブリーフに自分で書いて新規開始 |
| **Compact** | `/compact [指示]` | モデルに要約させ、その上で続行 |
| **Subagent** | Agent ツール委譲 | 別コンテキストで実行し結果だけ受け取る |

> **「修正するより巻き戻す」が最重要習慣**。失敗したアプローチをコンテキストに残したまま「やり直して」と頼むより、ファイル読み込み直後に巻き戻して「アプローチ A は使うな、直接 B」と再プロンプトする方が結果がよい（Thariq）。

### `/compact` vs `/clear`

| 観点 | `/compact` | `/clear` (+ 自分でブリーフ) |
| --- | --- | --- |
| 手間 | 低 | 中 |
| コンテキスト品質 | モデル依存（要約品質次第） | 自分が選別したものだけ |
| 残るもの | 要約 + コード/ファイル状態の重要分 | ブリーフだけ |
| 最適 | "この方向で続ける、ノイズだけ落としたい" | "別タスクへ移る、最小限だけ持っていきたい" |

`/compact` は指示で焦点を絞れる: `/compact focus on the auth refactor, drop the test debugging`。

`CLAUDE.md` に **"Compact Instructions"** セクションを書いて要約時に保持させる項目を指定できる（公式）:

```markdown
# Compact Instructions
- Always preserve the full list of modified files
- Always preserve the test commands used
- Drop debug log dumps
```

### `/rewind` の使い方

- `Esc Esc` または `/rewind` でメニュー表示
- 「会話のみ復元 / コードのみ復元 / 両方復元 / 選択メッセージから要約」が選べる
- すべての Claude のアクションは自動チェックポイント化されている（外部副作用は除く）
- セッションを跨いで永続化される

特に有用な使い方（Thariq）:

- ファイルを読んだ直後に巻き戻して「アプローチ A は使わない、直接 B に進んで」と再プロンプト
- "summarize from here" で過去の試行をハンドオフメッセージにして、新規セッションへ

## 1.3 「悪いコンパクト」を避ける

長いデバッグ後に autocompact が走り、その後「別件の警告も直して」と指示すると、要約から落ちて Claude が迷子になる。理由は **コンパクト時点で Claude は最も知能が低い状態**にあるから（Thariq）。

対策:

- 1M を活かして **コンパクト前に積極的に手動 `/compact "次は X をやりたい"`** を打つ
- `CLAUDE.md` に「Compact Instructions」を書く
- 自動コンパクトが複数回走る前にセッションを切る判断をする
- thrashing エラー（単一の巨大ファイル/出力でコンパクト後すぐ満杯になる）が出たら、Subagent に委譲するか出力をフィルタする

## 1.4 コンテキスト最適化 4 戦略 (WSCE)

[fabymetal](https://note.com/fabymetal/n/n3f0f2873b56c) が整理しているフレームワーク。公式の語彙ではないが実務指針として有用:

| 戦略 | 内容 | 適用例 |
| --- | --- | --- |
| **W**rite | 情報を構造化・圧縮して書く | 100 ページの仕様を要点 1 ページに |
| **S**elect | 必要な部分だけ選択的に読む | ファイル全体でなく該当セクションのみ `Read offset/limit` |
| **C**ompress | 冗長情報を落とす | ログから関連行だけ抽出して渡す |
| **I**solate | サブエージェントに分割 | 探索を Explore agent に委譲、結果だけ受け取る |

## 1.5 トークン hot spot の典型パターンと対策

agent-coach がトークン消費を分析するときに見るべき典型パターン。

### a. 巨大ファイルの全文 Read

- シグナル: 単一 Read で N 千行のファイル全体を取り込む
- 対策: `offset/limit` で必要セクションのみ、または `Grep` パターンマッチ
- 提案文例: 「ファイル X の全文 Read（〜YK 字）が観測されました。Grep '<シンボル>' → Read with offset で該当 50 行に絞ると K 字に削減できます」

### b. 同じ Bash コマンドの繰り返し

- シグナル: `ls`, `cat`, `wc -l` 等が同じ引数で複数ターン
- 対策: 1 回の集約コマンドへ
- 提案文例: 「`ls A; ls B; ls C` を `ls A B C` 1 コールにまとめると tool_use 1 つで済みます」

### c. サブエージェント未活用な調査

- シグナル: 親が 5+ ファイル / 数十回の Read を行っている探索フェーズ
- 対策: `Agent(subagent_type=Explore)` 委譲。読取専用 Haiku で実行され、結果だけ親に戻る
- 提案文例: 「auth フローの理解に N 回の Read（合計 K 字）が観測されました。Explore agent に委譲すると親コンテキストにはサマリのみ載ります」

### d. 5 分以上空いた連続ターン

- シグナル: assistant ターン間隔が 5 分超
- 対策: prompt cache の TTL は 5 分。空くと cache miss でリアル input トークンが膨らむ
- 提案文例: 「ターン X-Y 間で 7 分空いており cache miss が発生しています。`ScheduleWakeup` の delaySeconds を 270 秒以下にするか、まとめて作業する設計に」

### e. 同じツール出力が繰り返し挿入される

- シグナル: 巨大 tool_result（> 5,000 字）が複数ターン間隔で同内容
- 対策: 1 回読んだら必要部分を別ファイル / 別ノートにまとめて参照、または `summarize from here`

## 1.6 コンテキストロットの典型シグナル

agent-coach がコンテキストロットを検出するときに見るべきシグナル（[agent-coach SKILL.md より整理]）:

| シグナル | 意味 | 提案 |
| --- | --- | --- |
| 1 セッション > 50 ターン | ロット閾値接近の目安 | `/clear` 候補 |
| 後半で初期指示への参照消失 | 同じファイルを Read し直す等 | Plan モードで方針確定して実行を短く保つ |
| 同じ Read / Bash が繰り返される | 短期記憶喪失の兆候 | 結果を `MEMORY.md` 等に書き出して持ち回す |
| 巨大 tool_result が複数 | コンテキスト浪費 | Subagent 委譲、または出力フィルタ |
| 後半で指示違反が増える | CLAUDE.md ルールが埋もれる | ルールを冒頭へ移動、Hook で強制 |

> 重要: これらは目安にすぎない。長い設計議論や深いデバッグでは、ターン数や応答サイズが大きくても問題ないケースが多い。**シグナル単体で断定せず、複数シグナルが揃ったときに「可能性」として提案する**こと。

## 1.7 セッション境界の置き方

公式 + 補助記事を統合した実務指針:

- **新タスクは新セッション**が基本方針（公式・Thariq）
- グレーゾーン: 直前の機能のドキュメントを書く等、再読み込みコストが大きく知性要求が低いタスクは流用してよい（Thariq）
- 補正を 2 回以上同じ件で出した→`/clear` してより具体的な初期プロンプトでやり直す方が早い（公式 アンチパターン）
- セッションは git ブランチのように扱える（`/rename` で名前付け、`claude --continue/--resume`、`--fork-session` で分岐）
- 同じセッションを複数ターミナルから resume すると会話が混線する。並列作業には `--fork-session`

## 1.8 セッション間の状態継承

公式と nyosegawa を統合した運用パターン:

- **Git をブリッジに使う**: セッション終了時に記述的コミットメッセージで commit。`git log --oneline -20` が最も信頼性高い「何が起きたか」の記録（[nyosegawa](https://nyosegawa.com/posts/harness-engineering-best-practices-2026/)）
- **進捗ファイルは JSON 推奨**: モデルの不適切編集リスクが低い（nyosegawa）
- **長期プロジェクトはテストを「機能リスト」に**: ドキュメントは腐敗するがテストは腐敗しにくい（nyosegawa）
- **auto memory** (`MEMORY.md`) を能動的にキュレートする: 200 行 / 25KB 上限を意識
- **CLAUDE.md の "Compact Instructions"** で要約境界を制御

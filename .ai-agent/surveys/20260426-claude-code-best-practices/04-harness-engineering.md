# 04. ハーネスエンジニアリングの視点

> [nyosegawa](https://nyosegawa.com/posts/harness-engineering-best-practices-2026/) の論を中心に、「プロンプトでなく仕組みで品質を強制する」方向の知見をまとめる。横断テーマだが、agent-coach が「Hook 化を提案するか／プロンプト改善で済ませるか」を判断する根拠に使う。

## 4.1 主張: ハーネスがモデルより重要

> "同一モデルでハーネス変更時はスコア 22 ポイント変動、モデル交換時は 1 ポイント程度"

- エンジニアの仕事は「正しいコード生産」から「**エージェントが確実に正しいコード生産する環境設計**」へ移行
- ハーネスへの投資は **複利効果**: リンタールール 1 つ追加で以降すべてのセッションでそのミスを防止
- 「プロンプトエンジニアリング」より「**ハーネスエンジニアリング**」と呼ぶべき

## 4.2 リポジトリ衛生

### 置くべきもの

- 実行可能アーティファクト（コード、テスト、リンタ設定、型定義、CI 設定）
- **Architecture Decision Records (ADR)**: ステータス（Accepted / Superseded / Deprecated）を時系列管理

### 置くべきでないもの

- 説明文書、設計概要、手書き API 説明（**腐敗リスク高**）
- 「エージェントにとって、リポジトリ内で発見できる古い情報は最新の真実と区別できない」

### 腐敗対策

- **テストはドキュメントより腐敗に強い**。「この機能はこう動く」を仕様文書ではなくテストで表現する
- ADR は古いものも残るが、Status フィールドで意図が読み取れる
- 進捗ファイル / TODO 系は **JSON 推奨**（モデルの不適切編集リスクが低い）

## 4.3 決定論的ツールへの委譲

> "LLM は従来リンタより高価で遅い。決定論的ツール使える場面では常にそちらを使うべき"

### 言語別推奨スタック (2026 年 3 月時点)

| 言語 | ツール | 理由 |
| --- | --- | --- |
| TypeScript / JS | Oxlint + Biome | ESLint 比 50〜100 倍高速、Shopify で 75 分 → 10 秒 |
| Python | Ruff | Flake8 等を統合、大規模で秒以下 |
| Go | golangci-lint | 50+ リンタ並列、キャッシュ対応 |
| Rust | Clippy (pedantic + allow_attributes 禁止) | allow 句でのルール回避を構造的に不可能化 |

### フィードバック速度の階層

1. **PostToolUse Hook (ms)** — フォーマッタ自動実行
2. **プリコミットフック (s)** — リント・型チェック
3. **CI/CD (min)** — 全テスト
4. **人間レビュー (h)** — コードレビュー

> ハーネスエンジニアリングの目標は **できるだけ多くのチェックをより速いレイヤーに移動すること**。

### PostToolUse の威力

- フィードバック注入: エラーメッセージを修正指示として機械的に注入
- フォーマッタ自動実行で 40〜50% の問題を自動解決、残りのみ Claude に注入
- マッチャー `Write|Edit|MultiEdit` でファイル操作すべてに適用

## 4.4 Hook 4 パターン（再掲）

| パターン | イベント | 役割 |
| --- | --- | --- |
| Safety Gates | PreToolUse | 破壊的コマンド・機密ファイル編集を Exit 2 でブロック |
| Quality Loops | PostToolUse | リント結果を `additionalContext` として注入、自己修正駆動 |
| Completion Gates | Stop | テスト通過まで完了許可しない |
| Observability | 全イベント | 意図・結果・コンテキスト損失を監視パイプライン送信 |

## 4.5 カスタムリンター戦略

Factory.ai の 4 カテゴリ分類が参考になる:

1. **検索容易性** — ファイル名規則、シンボル命名規則
2. **配置予測可能性** — ディレクトリ構造、レイヤー境界
3. **アーキテクチャ境界** — モジュール依存制約
4. **セキュリティ** — 危険 API 使用禁止

### エラーメッセージの書き方

> エラーメッセージは「何が間違っているか」だけでなく、「**具体的な修正手順 / コード例 / ADR 参照**」を記述する

例:

```text
[error] auth.ts:42  Forbidden: direct DB access in auth layer
  Auth layer must call DataAccessLayer (see ADR-0007).
  Fix: import { getUserById } from '@/dal/user' and replace `db.query(...)` with `getUserById(id)`.
  ADR: docs/adr/0007-auth-layer-isolation.md
```

### リンタ設定の保護

PreToolUse で `.eslintrc` `.prettierrc` `tsconfig.json` 等の編集をブロック。エージェントが「ルール違反 → ルールを緩めて回避」する逃げ道を塞ぐ。

## 4.6 AI 生成コード固有のリント違反

[nyosegawa] が指摘する典型:

| 違反 | 内容 | 検出方法 |
| --- | --- | --- |
| TypeScript `any` 乱用 | 型推論失敗時に逃げる | `@typescript-eslint/no-explicit-any` |
| コード重複 | 既存コードを検索せず新規生成 | jscpd / 構造重複検出 |
| ゴーストファイル | 既存ファイル修正せず新規作成 | PreToolUse hook で「同名近似ファイルが既に存在」を検出 |
| セキュリティ脆弱性 | AI 生成コードの 36〜40% に脆弱性含有 | semgrep / CodeQL |

## 4.7 ユニバーサル E2E テスト戦略

> エージェントは自分が書いたコードを「見る」手段がなければコンパイル通過だけで完了宣言する

**アクセシビリティツリー = ユニバーサルインターフェース**。Web / Mobile / CLI / Desktop / API すべてに適用可能な構造化テキスト出力の優先（スクリーンショットは補助）。

### Web

| ツール | トークン効率 | 用途 |
| --- | --- | --- |
| Playwright MCP | 114K/タスク | テスト生成 + Planner/Healer |
| Playwright CLI | 27K/タスク | 主力 E2E テスト |
| agent-browser (Vercel) | 5.5K/6 テスト | 最高効率（ref 方式でセレクタ脆さ回避） |

### モバイル / iOS

- Xcode 26.3 の MCP ネイティブサポート（2026 年 2 月）
- XcodeBuildMCP + Xcode 内直接動作

### CLI / TUI

- bats-core（Bash 自動テスト、TAP 出力）
- pexpect / expect（対話的操作制御）

### API / バックエンド

- **Hurl**（プレーンテキスト HTTP テスト、エージェント相性最高）
- Pact（マイクロサービス契約テスト）

### インフラ

- terraform test（ネイティブ）
- Conftest + OPA（決定論的ポリシーチェック）
- PreToolUse で本番 apply 禁止

### AI / ML パイプライン

- データ品質: GX Core / Soda Core / dbt Tests
- モデルベンチマーク: lm-evaluation-harness / LightEval
- LLM 評価: DeepEval / promptfoo / RAGAS
- 安全性: PyRIT / Constitutional Classifiers

## 4.8 計画と実行の分離（ハーネス観点）

エージェントの「完了」宣言は不正確: **E2E テスト通過を完了判断の条件化**することで信頼性が上がる。

タスク粒度のガイダンス:

- 「一度にすべてをやろう」傾向に対し明示的に「**一度に一つ機能**」と指示
- 大目標を小さい構成要素に分解、各完了後に次へ

## 4.9 セッション間状態管理（ハーネス観点）

### 起動ルーチン標準化

1. 作業ディレクトリ確認
2. Git ログ・進捗ファイル読み取り
3. 機能リストから次タスク選択
4. 開発サーバー起動・疎通テスト

### Git をブリッジに

- セッション終了時に記述的コミットメッセージで commit
- `git log --oneline -20` が最も信頼性高い「何が起きたか」の記録

## 4.10 プラットフォーム比較: Claude Code vs Codex

参考のため。本リポジトリは Claude Code 主だが、エコシステムの動向として:

| 面 | Claude Code | Codex |
| --- | --- | --- |
| 形態 | 作業場型（ローカル直接実行） | 密室型（クラウドサンドボックス） |
| 並列性 | シーケンシャル | 非同期並列実行可能 |
| Hooks | 全ツール対応（安定版） | Bash 限定（実験的、v0.117.0 導入） |
| コンテキスト管理 | PreCompact 保護、MCP Tool Search | サンドボックス分離 |

ハイブリッド戦略（nyosegawa）:

- **Claude Code で計画・設計・ハーネス構築**
- **Codex で並列実行・スケール**
- 共通レイヤー: AGENTS.md / CLAUDE.md / Skills / MCP / ADR / リンタ設定

## 4.11 最小実行可能ハーネス (MVH) の段階導入

### Week 1

- AGENTS.md / CLAUDE.md（50 行以下）
- プリコミットフック
- PostToolUse 自動フォーマット
- 初回 ADR 作成

### Week 2-4

- エージェントミスのたびにテスト追加
- 計画→承認→実行ワークフロー
- E2E テストツール導入
- Stop Hook でテスト通過を条件化

### Month 2-3

- カスタムリンター構築（修正指示つき）
- ADR とリンタールール紐づけ
- ドキュメント除去 → テスト化
- PreToolUse 安全性ゲート

### Month 3+

- ガベージコレクション
- 複数エージェント同時実行
- ハーネス効果の定量測定

## 4.12 アンチパターン（nyosegawa）

1. **プロンプトのみ依存** — 「テスト書いてね」では不十分。Hook で強制
2. **ドキュメント蓄積** — 説明文書は腐敗。型定義・テストで代替
3. **巨大な CLAUDE.md** — 1000 行超は最悪。50 行以下を目指す
4. **エージェント専用インフラ** — 優れた開発者インフラを作ればエージェントは自動恩恵
5. **ハーネス無しでスケール** — 複利的レバレッジでなく複利的認知負債が生まれる

## 4.13 agent-coach がハーネス観点で出すべき提案

agent-coach が transcript で「同じミスが繰り返されている」「ルールが守られていない」を見つけたとき、**プロンプト改善より Hook 化を優先提案**する基準:

| 状況 | 改善提案 |
| --- | --- |
| 同じスタイル違反が 3 回以上 | PostToolUse でフォーマッタ自動実行を提案 |
| 危険コマンドが実行直前で気付かれた | PreToolUse safety gate を提案 |
| テスト未実行で「完了」宣言 | Stop Hook completion gate を提案 |
| CLAUDE.md にあるルールが守られていない | CLAUDE.md → Hook に格上げを提案 |
| 同じファイル種類の編集後に手動チェックが入る | PostToolUse 自動チェック |

逆に Hook 化が **過剰**なケース:

- 1 度きりのタスクのため
- ユーザー判断が必要（「これでいい？」と聞く類のもの）
- 影響範囲が小さい（typo 修正等）

判断は「**この検出を毎回起こすことがプロジェクトの利益になるか**」で決める。

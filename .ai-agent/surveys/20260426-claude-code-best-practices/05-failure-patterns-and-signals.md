# 05. アンチパターンと transcript 上の検出シグナル

> agent-coach が「何を見て何を提案するか」のリファレンス。公式が名指ししているアンチパターン + 補助記事の知見 + transcript で観察可能なシグナルを統合して、検出 → 原因 → 提案のテンプレを提供する。

## 5.1 公式が名指ししているアンチパターン

[Best practices](https://code.claude.com/docs/en/best-practices) の "Avoid common failure patterns" より:

| パターン | 症状 | 公式推奨 fix |
| --- | --- | --- |
| **キッチンシンクセッション** | 1 セッションで複数無関係タスク | `/clear` を間に挟む |
| **永遠に修正** | 補正→失敗→再補正のループ | 2 回失敗で `/clear`、学んだ前提を入れた具体プロンプトで再開 |
| **過剰仕様の CLAUDE.md** | 重要ルールがノイズに埋もれて無視される | 行ごとに「消したら困るか？」で剪定。Hook に置換できるなら Hook 化 |
| **検証ギャップ** | それっぽいが動かない実装 | テスト/スクリプト/スクショを必ず添える、無理なら出荷しない |
| **無限探索** | "調査して" でファイル数百読み出す | スコープを切るか Subagent に委譲 |

## 5.2 transcript からの検出シグナル

### a. キッチンシンクセッション

| 検出 | 意味 |
| --- | --- |
| 1 セッション内で `cd` / 異なるディレクトリの `Glob` パターンが 3 回以上切り替わる | 別プロジェクトをまたいでいる可能性 |
| トピックを示すユーザー発話（タスク開始の宣言）が 3 回以上 | 複数タスクが混在 |
| 序盤の Read ファイルと終盤の Read ファイルがまったく別領域 | スコープ拡散 |

**提案**: 「このセッションは 3 つの異なるタスク（A, B, C）を含んでいるように見えます。各タスクの境界で `/clear` を挟むと、後半の精度が上がる可能性があります」

### b. 永遠に修正（loop of corrections）

| 検出 | 意味 |
| --- | --- |
| 同一トピックで 3 ターン以内に再指示が 2 回以上 | Claude が誤解し続けている |
| `not that` / `instead` / `違う` / `そうじゃなくて` 等の否定語が短いユーザーメッセージで連続 | 補正ループ |
| Claude の応答に "Wait, you mean..." が出ている | 解釈の不確実性 |

**提案**: 元プロンプト + Claude の解釈 + ユーザー修正をセットで提示し、書き換え案を出す。「補正が 2 回以上続いた時点で `/clear` してこの書き換え案で再開する方が効率的だった可能性があります」

### c. 過剰仕様の CLAUDE.md

| 検出 | 意味 |
| --- | --- |
| CLAUDE.md にあるルールに違反する Claude 行動 | 埋もれている |
| CLAUDE.md > 200 行 | 構造的にリスク（IFScale 研究） |
| 同じルールについてユーザーが繰り返し指示している | CLAUDE.md だけでは効いていない |

**提案**:

- 短くする（行ごとに「消したら困るか？」）
- 重要なものは冒頭へ移動 / IMPORTANT 強調
- 「絶対」型のルールは Hook 化を検討

### d. 検証ギャップ

| 検出 | 意味 |
| --- | --- |
| Edit/Write 後にテスト・型チェック・ビルドの実行が無い | 検証していない |
| Claude が「実装完了」と宣言したが続くターンでバグ報告がある | 自己検証されてない |
| 検証コマンドの `Bash` 呼出が 0 回 | 検証手段未提供 |

**提案**: 「このタスクでは Edit 後に検証が走っていません。次回のプロンプトに『実装後 `npm test src/auth.test.ts` を実行』を追加すると、Claude が自己検証できます」

### e. 無限探索

| 検出 | 意味 |
| --- | --- |
| 1 ターンで N (= 10+) ファイルの Read | スコープ過大 |
| `Grep` パターンが極端に広い（`.*` 等） | 範囲限定なし |
| 探索フェーズが全体の 50% 以上 | 親コンテキスト浪費 |

**提案**: 「ファイル探索に N トークン消費しています。次回は `Agent(subagent_type=Explore)` に委譲すると親コンテキストにはサマリのみ載ります」

2. **繰り返し修正** — 2 回失敗後 `/clear` で再スタート

## 5.4 検出と提案の優先順位

agent-coach が transcript を見るとき、提案の重大度は次の順で評価する:

### 重大度: High（即座に提案すべき）

- **危険コマンド検出** — Hook safety gate に格上げを提案
- **検証なしの「完了」宣言** — Stop Hook + テスト追加を提案
- **同じミスが 3 回以上反復** — 仕組み（Hook / リンタ）化を提案
- **シークレット流出**（`sk-...`, `Bearer ...`, `password=...`） — 即マスク + 流出可能性をユーザーに警告

### 重大度: Medium（次のセッション改善のため）

- 補正ループ（2 回以上の re-correction）
- CLAUDE.md ルール違反
- 巨大ファイル全文 Read
- スキル未活用
- description 不一致による誤起動

### 重大度: Low（運用改善のため）

- ターン数過多（> 50）
- 同じ Bash の繰り返し
- ターン間隔 5 分超（cache miss）
- MCP / アクティブツール過多

## 5.5 検出時の判断は慎重に

[agent-coach SKILL.md](../../../plugins/agent-coach/skills/agent-coach/SKILL.md) でも強調されているが、以下の点は誤検出を避けるために重要:

- **長い設計議論や深いデバッグ**ではターン数や応答サイズが大きくても問題ないケースが多い
- **探索フェーズで意図的に曖昧プロンプト**を使っているケースは正しい使い方
- **ユーザーが対話的にスペックを固めている**ときの短い往復は正常
- **同じファイルを Read し直す**のがコンテキストロットでなく、ファイルが変わったから再読込している場合もある

→ **シグナル単体で断定せず、複数シグナルが揃ったときに「可能性」として提案する**こと。トーンは「この可能性があります」「次のセッションで試してみる価値があります」に統一。

## 5.6 透明性ある報告のテンプレ

agent-coach のレポートで、各検出について以下を明示する:

1. **検出シグナル**（具体的にターン番号 / 行番号）
2. **背景となるベストプラクティス**（公式 or 記事の引用 + リンク）
3. **可能性の高い原因**（複数候補があれば全部）
4. **改善案**（具体的なプロンプト書き換え / 設定変更）
5. **誤検出の可能性**（"ただし、〜の場合は問題ありません" の脚注）

例:

```markdown
### 観点 4: コンテキストロットの可能性

**シグナル** (ターン 47 / `/Users/.../session-abc.jsonl:1234`)
- 同じファイル `src/auth/login.ts` を 5 回 Read している
- 後半（ターン 40 以降）で初期 CLAUDE.md ルール「テスト後にコミット」が無視されている
- セッション内ターン数が 62

**根拠**
- 公式 Best practices: "When the context window is getting full, Claude may start 'forgetting' earlier instructions"
  https://code.claude.com/docs/en/best-practices
- 30〜40 万トークンで劣化開始の知見（Thariq）

**可能性の高い原因**
- コンテキスト消費が劣化閾値を超えている

**改善案**
- 区切りで `/clear` を挟む（次タスクが「ドキュメント書き」程度なら新セッション）
- 長期記憶は CLAUDE.md でなく `MEMORY.md` 経由の auto memory に
- 探索は `Agent(subagent_type=Explore)` 委譲

**誤検出の可能性**
- ファイルが頻繁に書き換えられているなら再 Read は正常。Write 結果と内容が異なる場合は問題なし。
```
